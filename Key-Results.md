# Key Result
An _key result_ is high-level goal that ladders up to a defined objective; it is a large unit of work that is meant to be decomposed into smaller units of work. A key result should be specific, measurable, and attainable within a quarter. Often key results are the output of quarterly planning sessions. 

## Metadata

**Work Item Type**: `Key Result`

### Required

- **Title**: clear, concise, and descriptive.
- **Area Path**: At minimum, ensure the key result falls within a team's area path i.e., `Black Lagoon\[Team]`. 
- **Iteration**: The iteration should be set to the backlog of the team responsible for the key result. E.g., `Black Lagoon\[Team]`.
- **Description**: At minimum, some sort of problem statement should be defined for the Key result. **Ideally**, teams will want to detail personas involved, associated dependencies, acceptance criteria, and any other amplifying information pertaining to the key result.
- **Risk to Delivery**: The table below outlines how to label risk to delivery. 

|**Delivery Risk Level** | **Description** |
|--                      |               --|
| <font color='red'>High</font>      | Difficult to deliver with current resourcing and/or available technology. Follow on business decision required. |
| <font color='yellow'>Medium</font> | Some inhibiting factors exist, but risk mitigated at BL team level.|
| <font color='green'>Low</font>     |Able to deliver with current resources and/or available technology, no additional actions required.|
- **Links**: Ensure that the key result is child to an objective.



### Optional
- Any tags
